# Phase 2 Plan: `langgraph-ts-template` targeted refactor

## Goal

Remove `maw-cli` from the workflow template's dependency chain and shrink the scaffold contract to workflow-owned files only (`graph.ts` + `config.json`). Introduce the `prompts.global` / `prompts.agents` config shape, update the template engine and graph runtime to read from `maw.json` + the workflow-local `config.json`, and align all tests to the Phase 1 architecture.

## Decisions Cleared

- Generated target-project `graph.ts` must call `createGraph({ workflow: '__WORKFLOW_ID__' })`. The Phase 0 init plan example is updated in this revision to match.
- `src/scaffold/assets/config.json` is the single source of truth for embedded workflow prompt defaults. `DEFAULT_WORKFLOW_CONFIG` must be derived from that asset instead of duplicating the literal defaults in code.
- `loadProjectConfig()` falls back to `DEFAULT_PROJECT_CONFIG` only when `maw.json` is missing. If `maw.json` exists but cannot be read, parsed, or validated, the workflow runtime must throw.
- `WorkflowConfig` is optional and partial. Missing keys inherit defaults via deep merge.
- Explicit empty arrays in workflow prompt config also inherit defaults. Example: `prompts.global: []` keeps the embedded global defaults instead of clearing them.
- Invalid workflow-local prompt config or missing configured snippets must emit a loud warning and fall back to embedded defaults instead of breaking the workflow.

## Execution Notes

### `WORKFLOW_ID` derivation
Derive `WORKFLOW_ID` from `WORKFLOW_PACKAGE_NAME` by stripping any `@scope/` prefix:
```ts
const WORKFLOW_ID = WORKFLOW_PACKAGE_NAME.replace(/^@[^/]+\//, '');
```
For `langgraph-ts-template` this produces `langgraph-ts-template`. For `@org/coding` it produces `coding`. Template forks inherit the correct ID automatically when they rename their package.

### New scaffold contract
`maw-cli init` validates exactly these fields from a package's `./scaffold` export:
- `scaffold.packageName: string`
- `scaffold.workflow: string`
- `createScaffoldFiles: () => { 'graph.ts': string; 'config.json': string }`

Remove `directories`, `gitignore`, `rules`, and `assets` from the scaffold object entirely — `maw-cli` no longer reads them. The old types `ScaffoldAsset`, `ScaffoldRules`, `MawScaffold`, and the constants `SCAFFOLD_DIRECTORIES`, `SCAFFOLD_GITIGNORE`, `SCAFFOLD_RULES` should be deleted.

`readScaffoldAsset()` may remain only if it is still needed internally to derive `DEFAULT_WORKFLOW_CONFIG` from the scaffold asset. It is not part of the `maw-cli init` handoff contract and should not drag the old scaffold surface forward.

### `graph.ts.template` dual-token substitution
Add a second token `__WORKFLOW_ID__` alongside the existing `__WORKFLOW_PACKAGE__`. Both are substituted in `createScaffoldFiles()`. The generated target-project file must become:
```ts
import { createGraph } from 'coding';
export const graph = createGraph({ workflow: 'coding' });
```
`__WORKFLOW_PACKAGE__` → package name (the import), `__WORKFLOW_ID__` → workflow id (the runtime config path lookup). `createScaffoldFiles()` takes no arguments and reads `WORKFLOW_PACKAGE_NAME` / `WORKFLOW_ID` directly.

### `project-context.njk` deletion and `general-coding.njk` rename
`project-context.njk` (`Workspace path: {{ workspacePath }}`) is deleted. OpenViking owns project context retrieval; a static workspace-path injection conflicts with that role.

`general-coding.njk` is renamed to `general.njk` to match the snippet name declared in the new `config.json` format (`"general"`). Any test that previously referenced `'general-coding'` as a snippet name must be updated to `'general'`. Any test that previously referenced `readScaffoldAsset('ov')` or assertions about `project-context` content must be removed.

### `WorkflowConfig` shape
```ts
// src/config.ts
export type WorkflowConfig = {
    prompts?: {
        global?: readonly string[];
        agents?: Readonly<Record<string, readonly string[]>>;
    };
};

export type ResolvedWorkflowConfig = {
    prompts: {
        global: readonly string[];
        agents: Readonly<Record<string, readonly string[]>>;
    };
};
```
`WorkflowConfig` is the on-disk or caller-provided override bag. Empty objects and partial objects are valid. `ResolvedWorkflowConfig` is the normalized form used by the runtime after defaults are applied.

Merge rules:
- Missing keys inherit defaults.
- Explicit empty arrays also inherit defaults.
- Non-empty arrays replace the inherited value for that slot.
- Unspecified agents inherit their default arrays unchanged.

Composed order remains `[...global, ...agents[agent]]`.

### `DEFAULT_WORKFLOW_CONFIG`
Replaces the old `createConfig()`. Do not hand-write the literal defaults in `config.ts`. Parse `src/scaffold/assets/config.json` and export the resolved result as `DEFAULT_WORKFLOW_CONFIG` so the scaffolded file contents and embedded runtime defaults cannot drift.

### `TemplateComposition` in `composition.ts`
The template engine should operate on resolved prompt data only. `TemplateComposition` stays the normalized `{ global, agents }` shape:
```ts
export interface TemplateComposition {
    readonly global: readonly string[];
    readonly agents: Readonly<Record<string, readonly string[]>>;
}

export const resolveSnippets = (cfg: TemplateComposition, agent: string): string[] => {
    const agentSnippets = cfg.agents[agent];
    if (!agentSnippets) throw new Error(`Unknown prompt agent: ${agent}`);
    return [...cfg.global, ...agentSnippets];
};
```
Remove `AgentComposition` — per-agent entries are now plain `string[]`, not wrapper objects. The deep merge from partial `WorkflowConfig` into resolved `TemplateComposition` belongs in `config.ts` / `graph.ts`, not in `composition.ts`.

### Config split: exported vs internal
- `WorkflowConfig`, `ResolvedWorkflowConfig`, `workflowConfigSchema`, `parseWorkflowConfig`, `loadWorkflowConfig`, `resolveWorkflowConfig`, and `DEFAULT_WORKFLOW_CONFIG` — exported from `./config`; this is the only workflow-owned config schema
- Project config (`maw.json` shape) — internal to `graph.ts` only; never exported from the package; `maw.json` is `maw-cli` territory

### Runtime config loading in `graph.ts`
`loadRuntime(cfg)` must follow this chain:

1. **Project config**: try to read `maw.json` from `root`.
   - Missing file → use `{ workspace: '.', templates: { customPath: '.maw/templates' } }`
   - Existing but unreadable / invalid file → throw
2. **Workflow config** (priority order):
   - If `cfg.workflowConfig` is provided → deep-merge it into `DEFAULT_WORKFLOW_CONFIG` with `resolveWorkflowConfig()` (no I/O)
   - Else if `cfg.workflow` is provided → try `root/.maw/graphs/<workflow>/config.json`
     - Missing file → `DEFAULT_WORKFLOW_CONFIG`
     - Existing but invalid file → loud warning + `DEFAULT_WORKFLOW_CONFIG`
     - Valid file → deep-merge it into `DEFAULT_WORKFLOW_CONFIG`
   - Else → `DEFAULT_WORKFLOW_CONFIG`
3. **Agent selection**: `cfg.agent ?? first key of resolvedWorkflowConfig.prompts.agents`
4. **Prompt composition fallback**:
   - Try the resolved workflow config first
   - If composition fails because a configured snippet cannot be resolved, emit a loud warning and retry once with `DEFAULT_WORKFLOW_CONFIG`
   - If the embedded default snippets fail, rethrow; that is a package bug, not user config

`loadProjectConfig` should be a plain internal `async` function in `graph.ts` — not exported.

### Template engine `sources` simplification
The old `config.templates.sources` array is gone. The new engine always:
1. Checks `customPath` resolved against `root` — silently skips if the directory does not exist
2. Appends `templateDir` (embedded defaults) — always last, always present

Delete `repoDirs()` and the `'git'` branch entirely. Drop the `strict` option from `CreateTemplateEngineOptions`.

### `workspace` in the engine
`project-context.njk` is deleted, so no default template renders `{{ workspacePath }}`. Keep `workspace?: string` in `CreateTemplateEngineOptions` as a forward-compat default for custom templates that may reference `{{ workspacePath }}`. Set the bag entry to `opts.workspace ?? ''`. `graph.ts` passes `workspace` from the loaded project config. If a user-defined `.maw/templates/my-snippet.njk` uses `{{ workspacePath }}`, it will resolve correctly without additional wiring.

### `templateDir` export from scaffold
Export `templateDir` from `src/scaffold/index.ts` using the same dual-path pattern as `defaultDir()` in `engine.ts`:
```ts
const resolveTemplateDir = (): string => {
    const local = fileURLToPath(new URL('../templates/defaults', import.meta.url));
    if (existsSync(local)) return local;
    return fileURLToPath(new URL('../../src/templates/defaults', import.meta.url));
};
export const templateDir = resolveTemplateDir();
```
This is the path `maw-cli prompt:preview` (Phase 3) will use to locate embedded snippets.

### `graph.name` assignment
`cfg.config?.graph.name` no longer exists. New assignment:
```ts
graph.name = cfg.name ?? cfg.workflow ?? DEFAULT_GRAPH_NAME;
```

### `package-metadata.spec.ts` risks
- Line 9 executes `readFileSync(resolve(root, 'src/bin.ts'), 'utf8')` **at module-load time** — before any `it()` runs. Deleting `bin.ts` without updating this file causes every test in the file to fail at import. This update must be in the same step as `bin.ts` deletion.
- When `templateDir` becomes part of the scaffold contract, `package-metadata.spec.ts` must also keep asserting that `src/templates/defaults` stays in `package.json.files`. Phase 3 `prompt:preview` depends on those embedded templates being published.

### `lint:langgraph-json` + `lint:all`
`lint:langgraph-json` invokes `scripts/checkLanggraphPaths.js`, which validates the now-retired root-level `langgraph.json`. Both the script file and the npm script must be removed. Update `lint:all` to:
```json
"lint:all": "bun run lint && bun run format:check"
```

### `resolveEnvVars` removal rationale
The new workflow-local `config.json` contains only static snippet names — no `${VAR}` placeholders. `maw.json` also forbids interpolation per the init plan. `resolveEnvVars`, `envPattern`, `resolveValue`, and `hostEnv` have no remaining callers and should be deleted.
