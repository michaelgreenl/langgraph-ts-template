# langgraph-ts-template

[![CI](https://github.com/langchain-ai/new-langgraphjs-project/actions/workflows/unit-tests.yml/badge.svg)](https://github.com/langchain-ai/new-langgraphjs-project/actions/workflows/unit-tests.yml) [![Integration Tests](https://github.com/langchain-ai/new-langgraphjs-project/actions/workflows/integration-tests.yml/badge.svg)](https://github.com/langchain-ai/new-langgraphjs-project/actions/workflows/integration-tests.yml)

This template now ships the active Phase 6 MAW workflow-package contract.

The primary runtime surface is workflow-local `.maw/graphs/<workflow>/opencode.json` launched through `maw-cli dev <workflow>`. The default interactive workflow starts on visible primary `planner`, hands explicit execution to visible primary `manager`, and uses hidden `coder` subagents for scoped implementation work.

## What ships

- a `./scaffold` export that materializes `graph.ts` and raw `opencode.json`
- a workflow-owned `opencode.json` validator surface (`workflowOpencodeSchema`, `parseWorkflowOpencode`, `loadWorkflowOpencode`)
- packaged `.opencode/agents/{planner,manager,coder}.md` prompts plus frontmatter permission baselines
- a retained `createGraph()` compatibility runtime for the separate direct LangGraph smoke path

## Target-project contract

```text
package.json
.maw/
  graphs/
    langgraph-ts-template/
      graph.ts
      opencode.json
      langgraph.json
  ov.conf
  ovcli.conf
```

Rules:

- `opencode.json` is the single workflow-local prompt, context, permission, and model/provider surface
- there is no project-root `maw.json`
- `.maw/templates/`, Nunjucks prompt composition, and `maw-cli prompt:*` are retired from the active contract
- invalid edited `opencode.json` fails before `maw-cli dev <workflow>` launches

## Runtime model

From a target project:

```bash
bunx maw-cli init
bunx maw-cli dev langgraph-ts-template
```

Runtime selection:

- TTY: bundled opencode interactive runtime, initial agent `planner`
- explicit execute request: hand the approved workflow off to `manager`
- execution: `manager` delegates step work to hidden `coder` subagents and auto-commits clean results
- non-TTY: the same workflow launches in server-only mode for SDK harnesses

The retained compatibility path stays separate:

```bash
bunx @langchain/langgraph-cli dev --config .maw/graphs/langgraph-ts-template
```

That LangGraph path exists only for compatibility and `/runs/wait` smoke. It is not the primary interactive UX after Phase 6.

## Package author surface

Workflow authors usually change these files:

- `src/scaffold/assets/opencode.json` — default workflow-local runtime config
- `.opencode/agents/{planner,manager,coder}.md` — packaged prompts and permission baselines
- `src/scaffold/index.ts` — scaffold and validator exports
- `src/agent/graph.ts` — retained LangGraph compatibility runtime

The scaffold contract is intentionally small:

- `createScaffoldFiles()` returns only `graph.ts` and `opencode.json`
- `graph.ts` exists only for the retained compatibility path
- `langgraph.json` is generated by `maw-cli`, not by the workflow package
